# 项目规则（Claude Code 版）

本文件只定义硬规则、确认门槛、停止条件、留痕规则与文件纪律。

流程阶段划分见 `AI_WRITING_SOP.md`。
内容写法见 `CONTENT_FRAMEWORK.md`（核心）+ `CONTENT_FRAMEWORK_structure.md`（结构阶段起）+ `CONTENT_FRAMEWORK_writing.md`（Prototype 阶段起）。

## 1. 核心原则

### 1.1 不抢跑
Claude Code 只能推进到当前阶段的关键检查点，不能默认自动跑完全流程。

### 1.2 用户保留主导权
方向、取舍、结构、语气和是否继续，最终由用户拍板。

### 1.3 先收敛，再扩写
在方向、结构、Prototype 未确认前，不得直接生成长篇正文。

### 1.4 先过 Killer Test，再过前提因果
脑洞必须先通过 Killer Test（脑洞粗筛，满分 8 分，6 分通过）才能进入前提因果校验。每个脑洞最多重写 2 次，超过则 freeze。如果一个脑洞连"为什么非要这么做""为什么不能用更简单办法"都答不通，就不能靠裂变、钩子和文风建议硬往前推。

### 1.5 留痕优先
关键讨论、评估结论、结构决策和返工结论应尽量落成文件。

### 1.6 成立优先于放大
如果故事方向、结构、人物驱动或 Prototype 仍不成立，不得用继续扩写掩盖基础问题。

### 1.7 保留快节奏，但不纵容廉价刺激
允许开头快、冲突早、节奏紧；不允许空降信息、强行反转、工具人角色和提纲 prose。

## 2. 强制确认门槛

以下节点必须等待用户明确确认后才能继续：

1. 脑洞裂变后
2. 方向评估后
3. 结构方案后（含情绪曲线确认）
4. 详细大纲后
5. Scene Pressure Test 后
6. Prototype 后
7. 每次扩写后

如果用户没有明确确认，默认停在当前阶段。

## 3. 停止条件

遇到以下情况时，必须停止向下推进，并先处理问题：

1. 方向未定
2. 结构未稳
3. 关键信息缺失
4. 用户要求修改当前阶段
5. 日志或文件状态异常
6. Killer Test 未通过（0-3 分），或已达 2 次重写上限
7. 前提因果链不成立，尤其答不通"为什么非要这么做"和"为什么不能更简单"
8. 关键动机不清
9. 关键转折不成立
10. 人物明显工具化
11. Prototype 已露出明显提纲腔
12. Scene Pressure Test 关键场景低于 4 分
13. 扩写方式开始失真
14. 正文过平或过假
15. 用户反馈"太平""没劲""不够狠""不像盐选"——等同于质量未通过，不允许解释后继续用原版，必须按 CONTENT_FRAMEWORK_writing.md §6.4 提供替代方案

## 4. 日志规则

### 4.1 `01_Evaluation_Log.md` 必须追加
- 保留所有历史评估轮次
- 默认采用追加，不覆盖旧结论
- 每轮要有清晰轮次标记

### 4.2 `00_Creative_Chat_Log.md` 强制追加，版权留痕
- 每轮推进前必须记录本轮关键决策（`- 关键决策: <内容>`），否则 guard.py 将阻止 advance
- 记录方向、结构和回退决策，以摘要为主
- 这是版权留痕文件，记录创作过程，不得删除历史记录

### 4.3 修正采用补记，不采用抹除
如果前一轮判断被推翻，应追加修正说明，而不是删除历史。

### 4.4 每次沟通结果都要落成单独 md
只要当前轮次形成了可复用结果，就必须新增单独归档文件。

### 4.5 单轮归档默认 new-file-only
- 默认新增，不覆盖旧文件
- 需要修正上一轮时，也新增修正版，不覆盖历史

### 4.6 前提校验失败也要留痕
如果一个脑洞在前提因果校验阶段就失败了，应明确记录失败点和失败原因。失败样本本身就是流程优化输入。

### 4.7 回退不等于删除
回退默认只改变当前阶段，不抹除旧路径。

## 5. 文件纪律

### 5.1 根目录只放框架内容
根目录优先放框架说明、规则文档、模板和脚本。

### 5.2 实际创作产物集中到 `output/`
具体脑洞、评估、结构稿、Prototype 和正文草稿，统一放在 `output/<project>/`。

### 5.3 总文件与单轮文件职责分离
- 总文件保存当前阶段最新可读版本
- 单轮文件保存每轮独立留痕
- 只更新总文件、不补单轮 md，视为留痕缺口

### 5.4 扩写版本管理

- 初稿固定命名 `scene-xx.md`
- 每轮用户标注修改后，AI 输出递增版本：`scene-xx-v1.md`、`scene-xx-v2.md`...
- 版本号只递增，不覆盖旧版本文件
- 用户确认后保存为 `scene-xx-终稿.md`（干净正文，无标注痕迹）
- 全部终稿完成后合并为 `chapters/《小说标题》.md`
- 修改留痕格式：`##原文##用户意见##AI修改说明##`，写在新版本对应位置

### 5.5 状态文件负责流程硬约束
- `project_state.json` 负责记录当前阶段、确认状态、前提校验结果和回退历史
- 需要推进或回退时，优先通过 `scripts/guard.py` 修改状态

### 5.6 旧项目迁移按"下一轮开始"处理
- 不强制补建所有历史单轮文件
- 从当前继续推进的新轮次开始遵守 `stage_logs/` 规则

## 6. 内容风险纪律

Claude Code 在继续推进前，至少要检查这些风险：

1. 是否存在明显便利型推进
2. 是否存在关键转折无前置支撑
3. 是否存在角色只负责送信息、送背刺或送反转
4. 是否存在概述过量、对话同质化、短视频文案腔
5. 是否存在节奏掉速、局势不变、只有台词变狠

如果这些问题已经明显出现，应该回修当前层或上一层，而不是继续往后堆字数。

## 7. 输出纪律

1. 每次只解决一个层级的问题
2. 不把建议伪装成定稿
3. 不为了显得积极而超量生成
4. 受控试写必须明确标注
5. 若当前判断仍不稳定，应优先给出继续、补档、回修或回退建议
6. 如果当前目标是优化框架，应优先报告流程缺陷，而不是勉强把故事讲顺
