# 项目规则（Claude Code 版）

本文件只定义硬规则、确认门槛、停止条件、留痕规则与文件纪律。

流程阶段划分见 `AI_WRITING_SOP.md`。
内容写法见 `CONTENT_FRAMEWORK.md`（核心）+ `CONTENT_FRAMEWORK_structure.md`（Structure 阶段起）+ `CONTENT_FRAMEWORK_writing.md`（Prototype 阶段起）。

## 1. 核心原则

### 1.1 不抢跑
Claude Code 只能推进到当前阶段的关键检查点，不能默认自动跑完全流程。

### 1.2 用户保留主导权
方向、取舍、结构、语气和是否继续，最终由用户拍板。

### 1.3 先收敛，再扩写
在方向、结构、Prototype 未确认前，不得直接生成长篇正文，也不得把 `review_required` 直接硬改成 `ready`。

### 1.4 先过 Killer Test，再过前提因果
脑洞必须先通过 Killer Test（脑洞粗筛，满分 8 分，6 分通过）才能进入前提因果校验。每个脑洞最多重写 2 次，超过则 freeze。如果一个脑洞连“为什么非要这么做”“为什么不能更简单办法”都答不通，就不能靠裂变、钩子和文风建议硬往前推。

### 1.5 留痕优先
关键讨论、评估结论、结构决策和返工结论应尽量落成文件。

### 1.6 成立优先于放大
如果卡片、母题、平台执行或 Prototype 仍不成立，不得用继续扩写掩盖基础问题。该修就 `rework`，该停就 `hold`，该否决就 `reject`。

### 1.7 保留快节奏，但不纵容廉价刺激
允许开头快、冲突早、节奏紧；不允许空降信息、强行反转、工具人角色和提纲 prose。

## 2. 强制确认门槛

完整列表和可接受信号：`python3 scripts/guard.py rules`

核心规则：idea_fission / evaluation / structure / outline / scene_pressure_test / prototype / expansion 这 7 个阶段必须等用户明确确认后才能继续。用户没有明确确认，默认停在当前阶段。

接口层的补充规则：

- 卡片在 `raw_idea -> classified` 后，如果字段未齐、平台未定或约束冲突尚可补齐，先标 `review_required`
- 如果信息不足到无法判断，或者用户明确要求暂停，标 `hold`
- 如果当前层可以在同一条路线内修正，标 `rework`
- 如果当前路线本身不成立，或者继续下去会把问题放大，标 `reject`

## 3. 停止条件

完整 15 条停止条件：`python3 scripts/guard.py rules`

关键条目摘要：方向未定、结构未稳、关键信息缺失、日志或文件状态异常、Killer Test 未通过、前提因果链不成立、关键动机不清、关键转折不成立、人物明显工具化、Prototype 已露出明显提纲腔、Scene Pressure Test 关键场景低于 4 分、扩写方式开始失真、正文过平或过假、用户反馈“太平”“没劲”“不够狠”“不像盐选”。

处理方式：

- 进入 `hold`：当信息、确认或外部条件不足，当前阶段先暂停
- 进入 `rework`：当同一路线还能靠局部修正救回来
- 进入 `reject`：当当前路线不成立，继续推进只会浪费字数或放大失真

关键判断要和 `AI_WRITING_SOP.md` 保持一致，不把参考层、阶段层和卡片层混成一层。

## 4. 日志规则

### 4.1 `01_Evaluation_Log.md` 必须追加
- 保留所有历史评估轮次
- 默认采用追加，不覆盖旧结论
- 每轮要有清晰轮次标记

### 4.2 `00_Creative_Chat_Log.md` 强制追加，版权留痕
- 每轮推进前必须记录本轮关键决策（`- 关键决策: <内容>`），否则 guard.py 将阻止 advance
- 记录方向、结构和回退决策，以摘要为主
- 这是版权留痕文件，记录创作过程，不得删除历史记录

### 4.3 修正采用补记，不采用抹除
如果前一轮判断被推翻，应追加修正说明，而不是删除历史。

### 4.4 每次沟通结果都要落成单独 md
只要当前轮次形成了可复用结果，就必须新增单独归档文件。

### 4.5 单轮归档默认 new-file-only
- 默认新增，不覆盖旧文件
- 需要修正上一轮时，也新增修正版，不覆盖历史

### 4.6 前提校验失败也要留痕
如果一个脑洞在前提因果校验阶段就失败了，应明确记录失败点和失败原因。失败样本本身就是流程优化输入。

### 4.7 回退不等于删除
回退默认只改变当前阶段，不抹除旧路径。

## 5. 文件纪律

### 5.1 根目录只放框架内容
根目录优先放框架说明、规则文档、模板和脚本。

### 5.2 实际创作产物集中到 `output/`
具体脑洞、评估、结构稿、Prototype 和正文草稿，统一放在 `output/<project>/`。`IDEA_CARD.md` 是这里的正式输入锚点，`EXPANSION_CONTEXT.md` 只负责继承关系，不单独升级成新权威。

### 5.3 总文件与单轮文件职责分离
- 总文件保存当前阶段最新可读版本
- 单轮文件保存每轮独立留痕
- 只更新总文件、不补单轮 md，视为留痕缺口

### 5.4 扩写版本管理

- 初稿固定命名 `scene-xx.md`
- 每轮用户标注修改后，AI 输出递增版本：`scene-xx-v1.md`、`scene-xx-v2.md`...
- 版本号只递增，不覆盖旧版本文件
- 用户确认后保存为 `scene-xx-终稿.md`（干净正文，无标注痕迹）
- 全部终稿完成后合并为 `chapters/《小说标题》.md`
- 修改留痕格式：`##原文##用户意见##AI修改说明##`，写在新版本对应位置

### 5.5 状态文件负责流程硬约束
- `project_state.json` 负责记录当前阶段、确认状态、前提校验结果和回退历史
- 需要推进或回退时，优先通过 `scripts/guard.py` 修改状态

### 5.6 旧项目迁移按“下一轮开始”处理
- 不强制补建所有历史单轮文件
- 从当前继续推进的新轮次开始遵守 `stage_logs/` 规则

## 6. 内容风险纪律

Claude Code 在继续推进前，至少要检查这些风险：

1. `IDEA_CARD` 是否还停留在 raw_idea 的模糊状态，却已经被当成 ready 使用
2. `target_platform` 是否和当前正文基线一致，是否出现番茄 / 知乎 / 盐选写法混用
3. `primary_motif` 是否发生漂移，是否把题材壳当成了母题本体
4. `must_have` 与 `must_avoid` 是否被后文悄悄改写
5. 是否存在便利型推进、关键转折无前置支撑、角色只负责送信息 / 送背刺 / 送反转
6. 是否存在概述过量、对话同质化、短视频文案腔、节奏掉速、局势不变

如果这些问题已经明显出现，应该回修当前层或上一层，而不是继续往后堆字数。

边界口径如下：

- `review_required` 用在卡片或当前层仍需人工确认，但修补后可以继续的情况
- `hold` 用在信息、状态、确认或外部条件还不足，当前必须暂停的情况
- `rework` 用在同一条路线内还能针对性修正的情况
- `reject` 用在当前路线不成立、继续推进会把失真固定下来的情况

## 7. 输出纪律

1. 每次只解决一个层级的问题
2. 不把建议伪装成定稿
3. 不为了显得积极而超量生成
4. 受控试写必须明确标注
5. 若当前判断仍不稳定，应优先给出继续、补档、回修或回退建议
6. 如果当前目标是优化框架，应优先报告流程缺陷，而不是勉强把故事讲顺
